## Summary

Deliver the **first Phase 2 slice** (`PROJECT REQUIREMENTS.md` ¬ß11.2 Week 13‚Äì14): artist-facing analytics on the existing dashboard. Add a new NestJS **`analytics`** module with aggregated queries over existing `earnings` and `tracks` tables, plus client UI sections on `ArtistDashboardPage` using design-system cards and tables. One optional migration adds an index on `earnings(artist_id, created_at)` for timeline queries.

## Technical Context

**Language/Version**: TypeScript on Node.js 20+ (api + client workspaces)

**Primary Dependencies**:
- API: NestJS 11+, TypeORM (existing entities: `Earnings`, `Track`, `Artist`)
- Client: React 18, Vite, TanStack Query, Tailwind design system (003/004)
- Auth: `JwtAuthGuard` + `RolesGuard` + `@Roles(UserRole.ARTIST)` (mirror `tracks.controller.ts`)

**Storage**: PostgreSQL 15+ ‚Äî **no new tables**; migration `005-EarningsArtistIndex.ts` adds query index only

**Testing**: Manual VS-401‚ÄìVS-404 per `quickstart.md`; optional service unit tests for aggregation logic

**Target Platform**: Web SPA + REST API

**Project Type**: Yarn monorepo ‚Äî `api/` + `client/`

**Performance Goals**: Dashboard aggregate queries p95 < 300ms for ‚â§100 tracks per artist

**Constraints**:
- mako module layout under `api/src/modules/analytics/`
- Routes under `/api/v1/analytics/`
- Scope queries by `req.user['artistId']` from JWT
- Smallest diff ‚Äî no chart libraries; table/CSS bars for timeline

**Scale/Scope**: 1 new API module, 2 endpoints, 1 migration (index), 1 client hook, dashboard UI extension

**Reference**: `PROJECT REQUIREMENTS.md` ¬ß3.4, ¬ß6.3.1 `/analytics/*`, existing `payments/entities/earnings.entity.ts`
